## 1. API 调用稳定性

### 实验过程中观察到的现象

在 Haiku 和 Sonnet 的实验运行过程中，终端日志偶尔出现如下重试提示：

```
[Retry] API error (attempt 1/3): APITimeoutError: Request timed out or interrupted.
  → retrying in 2s
```

这是由第三方 API relay 的瞬时网络波动导致的请求超时。

### 框架内置重试机制

AutoChip 的 `src/llm/client.py` 实现了自动重试逻辑：

1. 每次 API 调用最多重试 **3 次**
2. 重试间隔按 2s → 5s → 10s 递增（指数退避）
3. 重试成功后，返回的结果与正常调用完全等价
4. 仅当 3 次重试全部失败后，才标记 `api_error=true` 并记入结果

### 数据完整性验证

对本轮三组正式实验的 `details.json` 进行了逐候选（candidate）级别的审计：

| 模型 | Candidate 级 API 错误数 | 结论 |
|------|----------------------|------|
| claude-haiku-4-5-20251001 | **0** | 数据完整 |
| claude-sonnet-4-6 | **0** | 数据完整 |
| gpt-5.4 | **0** | 数据完整 |

**所有 API 超时均在重试后成功恢复**，无任何候选因 API 错误而丢失或跳过。

## 2. 实验可重复性说明

### 实验参数

| 参数 | 值 |
|------|---|
| temperature | 0.7 |
| feedback_k | 3（每轮生成 3 个候选） |
| max_iterations | 5（最多 5 轮 feedback） |
| 编译器 | iverilog (SystemVerilog 2012) |
| 仿真器 | vvp |

### 随机性来源

由于 `temperature=0.7`，LLM 的生成具有随机性。这意味着：

- 同一模型在同一题上重复运行可能产生不同结果
- 个别题目的 pass/fail 结果可能在不同运行间翻转
- **总体通过率的趋势是稳定的**，但精确百分比可能有 ±1-2 题的波动

### 结论

本轮实验数据**无脏数据、无 API 导致的结果丢失**，可视为有效实验结果。实验中观察到的 API retry 日志仅为网络层容错机制的正常工作记录，不影响最终数据质量。

如后续需要更高统计置信度，可对关键模型进行 3 次重复实验（每次使用不同 random seed），取平均通过率和置信区间。
