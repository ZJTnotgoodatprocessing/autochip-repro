## 案例 1：float_multi — Feedback 修复强模型专属复杂设计

### 题目信息

| 属性 | 值 |
|------|---|
| 设计名 | float_multi |
| 类别 | Arithmetic/Other |
| 难度 | ★★★★★（最难） |
| 核心挑战 | IEEE 754 浮点乘法器：指数加法、尾数乘法、符号处理、规格化 |

### 三模型表现

| 模型 | Zero-shot | Feedback | Improved? |
|------|-----------|----------|-----------|
| Haiku 4.5 | ❌ FAIL (0.00) | ❌ FAIL (0.00) | No |
| Sonnet 4.6 | ❌ FAIL (0.00) | ✅ **PASS** (iter=1) | **YES** |
| GPT-5.4 | ❌ FAIL (0.00) | ✅ **PASS** (iter=1) | **YES** |

### 分析

**三个模型在 zero-shot 下全部失败**——IEEE 754 浮点乘法器的实现需要精确处理指数偏移量、尾数隐含位、结果规格化等多个细节，仅凭自然语言描述很难一次性生成正确实现。

**强模型通过 feedback 在仅 1 轮迭代内修复**——编译/仿真错误信息精确指出了失败点（如指数计算偏差或尾数截断错误），Sonnet 和 GPT 具备足够的推理能力在 feedback 引导下定位并修正这些错误。

**Haiku 即使有 feedback 仍无法修复**——说明 feedback 并非万能，底层模型的推理能力是前提条件。弱模型可能理解不了编译错误的根因，或者生成的修复代码引入了新错误。

### 支撑论文观点

- **有效性论证**：AutoChip 的 feedback 机制能将"所有模型 zero-shot 全部失败"的最难设计变为"强模型可通过"
- **模型差异论证**：Feedback 的收益与模型能力正相关——强模型能利用错误信息做定向修复，弱模型则不能
- **适用于论文第 X 章**：实验结果分析 → "Feedback 对强模型的增益"小节

## 案例 2：LIFObuffer — 弱模型可被 Feedback 拉起的典型题

### 题目信息

| 属性 | 值 |
|------|---|
| 设计名 | LIFObuffer |
| 类别 | Memory/LIFO |
| 难度 | ★★★（中） |
| 核心挑战 | 栈式存储结构，含读写指针管理、满/空标志判断 |

### 三模型表现

| 模型 | Zero-shot | Feedback | Improved? |
|------|-----------|----------|-----------|
| Haiku 4.5 | ❌ FAIL (0.00) | ✅ **PASS** (iter=1) | **YES** |
| Sonnet 4.6 | ❌ FAIL (0.00) | ❌ FAIL (0.00) | No |
| GPT-5.4 | ✅ PASS | ✅ PASS (iter=1) | - |

### 分析

**这是一个差异化模式的典型体现**：

- **GPT-5.4 zero-shot 即通过**——说明最强模型对栈结构的理解已足够生成正确代码
- **Haiku zero-shot 失败，但 feedback 1 轮修复**——仿真错误精确指出了指针逻辑或空/满判断的错误，Haiku 虽然能力较弱，但在 feedback 引导下仍能修复中等难度的设计
- **Sonnet 反而无法通过 feedback 修复**——说明 feedback 效果并非完全随模型能力单调递增，存在随机性因素（如 temperature=0.7 下的生成路径差异）

### 支撑论文观点

- **有效性论证**：Feedback 能将弱模型（Haiku）在中等难度任务上"拉起"到强模型的水平
- **AutoChip 核心价值**：对于资源受限的场景（只能使用轻量模型），feedback 机制提供了"以计算换准确度"的可行路径
- **适用于论文第 X 章**：实验结果分析 → "Feedback 对弱模型的补偿效应"小节

## 案例 3：sequence_detector — 所有模型均失败的天花板题

### 题目信息

| 属性 | 值 |
|------|---|
| 设计名 | sequence_detector |
| 类别 | Control/FSM |
| 难度 | ★★★（中） |
| 核心挑战 | 序列检测器，需根据描述推断正确的状态转移逻辑 |

### 三模型表现

| 模型 | Zero-shot | Feedback | Improved? |
|------|-----------|----------|-----------|
| Haiku 4.5 | ❌ FAIL (-1.00) | ❌ FAIL (-1.00) | No |
| Sonnet 4.6 | ❌ FAIL (-1.00) | ❌ FAIL (-1.00) | No |
| GPT-5.4 | ❌ FAIL (-1.00) | ❌ FAIL (-1.00) | No |

> rank = -1.00 表示编译失败（代码无法通过 iverilog 编译）

### 分析

**所有模型在 zero-shot 和 feedback 下均编译失败**。rank = -1.00 说明 LLM 生成的代码连编译都无法通过，这意味着问题不在于逻辑不对（仿真不通过），而在于生成代码的结构性错误——可能是 module 名不匹配、端口声明不一致、或者语法错误。

**这暴露了当前框架的一个局限性**：feedback 机制依赖编译/仿真错误信息来引导修复，但如果 LLM 反复生成结构性错误的代码（如持续用错 module 名），feedback 的信息量不足以引导模型跳出这个错误模式。

**可能的根因**：
1. RTLLM 描述中的 module 接口定义可能不够清晰，LLM 无法正确推断端口列表
2. 当前 `_extract_module_header()` 函数可能未能为 sequence_detector 正确解析出 module header
3. 编译错误信息可能被截断，未能传递足够的修复线索

### 支撑论文观点

- **局限性讨论**：AutoChip 框架在面对"LLM 持续生成结构性错误代码"的场景下效果有限
- **改进方向**：可考虑增加"prompt 修正"机制——当检测到连续编译失败时，自动调整 prompt 策略（如补充更明确的 module header）
- **适用于论文第 X 章**：局限性与未来工作 → "框架失败模式分析"小节

## 总结：三类案例的论文价值

| 案例类型 | 代表题 | 核心发现 | 论文位置 |
|---------|--------|---------|---------|
| Feedback 修复强模型 | float_multi | 强模型 + feedback 能解决最难设计 | 有效性论证 |
| Feedback 拉起弱模型 | LIFObuffer | 弱模型通过 feedback 追上强模型 | 成本效益论证 |
| 天花板题（全部失败） | sequence_detector | 编译失败模式下 feedback 失效 | 局限性讨论 |

三个案例共同构成了论文实验分析部分的"**有效 → 部分有效 → 无效**"递进叙事结构。
