# 第 3 章 系统设计与实现

本章介绍本文所构建的 AutoChip 风格 RTL 自动生成与反馈修复原型系统的总体架构、核心模块设计与关键实现细节。该系统以"生成—编译—仿真—反馈—修复"为核心闭环，支持多种 benchmark 接入、多模型切换、多种反馈策略与提示词策略，为后续实验提供了统一的可复现实验平台。

## 3.1 系统总体架构

### 3.1.1 设计目标

本系统的核心目标是构建一个可扩展、可审计的 AutoChip 风格实验平台，具体包括：（1）接入多种公开 RTL benchmark，将不同格式的题目统一为内部 Task 表示；（2）调用大语言模型生成 Verilog 代码，并通过 EDA 工具链进行编译与仿真验证；（3）将验证结果以结构化反馈形式返回模型，驱动迭代修复；（4）记录完整实验过程元数据，确保结果可追溯、可审计。

### 3.1.2 模块划分

系统由以下核心模块组成：

1. **任务加载器（Task Loader）**：负责从不同 benchmark 目录中读取题目描述、模块接口与测试平台文件，统一转化为内部 Task 数据结构。
2. **提示词构建器（Prompt Builder）**：根据任务描述、模块接口及可选的反馈信息，构建发送给大语言模型的提示词。支持多种提示策略和反馈粒度级别。
3. **大语言模型客户端（LLM Client）**：封装 API 调用逻辑，通过第三方统一网关接入多种模型，内置重试机制与错误分类。
4. **Verilog 提取器（Verilog Extractor）**：从模型返回的自然语言混合文本中提取 `module ... endmodule` 代码块。
5. **Verilog 执行器（Verilog Executor）**：调用 Icarus Verilog 编译器和 VVP 仿真器，执行编译与仿真，解析输出结果。
6. **评分器（Ranker）**：基于编译与仿真结果，按 AutoChip 定义的评分规则对候选代码打分。
7. **反馈循环控制器（Loop Runner）**：实现完整的迭代生成—验证—反馈闭环，管理候选生成、全局最优追踪与终止条件判断。
8. **实验产物管理器（Artifacts Manager）**：负责实验运行目录创建、元数据记录、结果序列化与 Git 元信息采集。

## 3.2 任务加载与 Benchmark 接入

### 3.2.1 Task 数据结构

为了屏蔽不同 benchmark 在文件组织和题目格式上的差异，系统定义了统一的 Task 数据结构，包含四个核心字段：任务名称（name）、自然语言描述（description）、模块接口声明（module_header）以及测试平台文件路径（testbench_path）。所有下游模块——提示词构建、代码生成、编译仿真——均仅依赖 Task 接口，不直接感知 benchmark 来源，从而实现了 benchmark 无关的处理流程。

### 3.2.2 VerilogEval-Human 加载器

VerilogEval 是 NVlabs 发布的 Verilog 代码生成评估基准，采用 spec-to-RTL 模式，提供自然语言描述和 TopModule 接口定义。其文件结构为每个题目提供独立的接口文件（`_ifc.txt`）、参考实现（`_ref.sv`）和测试平台（`_test.sv`）。

VerilogEval 的测试平台设计为同时实例化待测模块（TopModule）和参考模块（RefModule），通过逐周期比较两者输出来判断正确性。这意味着仿真时需要将参考实现与测试平台合并为一个文件。VerilogEval 加载器的核心逻辑是：读取自然语言描述作为 description，读取接口声明作为 module_header，将参考实现与测试平台合并后写入临时文件作为 testbench_path。

### 3.2.3 RTLLM 加载器

RTLLM 2.0 是香港科技大学发布的 RTL 生成评估基准，包含 50 个不同类别的硬件设计任务，按 Arithmetic、Control、Memory、Miscellaneous 四大类组织。与 VerilogEval 不同，RTLLM 的每个题目目录包含一个自然语言设计描述文件（`design_description.txt`）和一个独立的测试平台文件（`testbench.v`）。

RTLLM 与 VerilogEval 在以下方面存在显著差异：（1）RTLLM 的描述文件中嵌入了模块名称信息，需要通过正则表达式提取；（2）RTLLM 的测试平台直接实例化目标模块，无需合并参考实现；（3）RTLLM 的仿真输出格式不同——通过 `Your Design Passed` 或 `Test completed with N/M failures` 标记结果，而 VerilogEval 使用 `mismatched samples is N out of M` 格式。

RTLLM 加载器通过递归遍历分类目录来发现所有题目，对每个包含 `design_description.txt` 和 `testbench.v` 的目录创建一个 RTLLMProblem 对象。加载器还尝试定位参考实现文件（`verified_*.v`），虽然参考实现不参与自动评估流程，但可用于后续人工案例分析。

### 3.2.4 适配层的必要性

两个 benchmark 在文件结构、描述格式、测试平台组织和仿真输出格式上的差异，使得专用加载器成为必要。统一的 Task 接口确保了反馈循环控制器和评分器等下游模块无需关心数据来源，同时也使得系统在未来接入新的 benchmark 时只需新增加载器即可。

## 3.3 大语言模型调用与模型管理

### 3.3.1 统一 API 网关架构

本系统通过第三方统一 API 网关（API relay）接入多种大语言模型。网关的核心价值在于：保持 API 密钥和服务端点固定不变，仅通过模型名称参数切换不同的底层模型。这意味着从 Claude Haiku 切换到 GPT-5.4 只需修改一个环境变量，无需更改代码或密钥配置。

系统的模型选择遵循三级优先级：首先检查 `LLM_MODEL` 环境变量（推荐的提供商无关名称），其次检查 `ANTHROPIC_MODEL`（保持向后兼容），最后回退到默认模型（claude-haiku-4-5-20251001）。这一设计使得批量实验脚本可以通过命令行参数 `--model` 方便地切换模型。

### 3.3.2 重试机制与错误分类

大规模实验（如 STUDY_12 × 3 模型 × 多条件）涉及数百次 API 调用，API 的瞬态错误（如 HTTP 503、连接超时、中转站负载均衡临时不可用）不可避免。系统内置了指数退避重试机制：对可重试的瞬态错误（HTTP 429/500/502/503/504/529、连接错误、超时错误）自动重试最多 3 次，退避间隔为 2s、5s、10s。

当所有重试均失败时，系统不会抛出异常中断整个批量实验，而是将错误信息封装为结构化的 APIError 对象，包含错误类型标签（如 `http_503`、`timeout`、`model_not_found`、`connection_error`）和原始异常信息。该错误会被记录到该候选结果的 `api_error` 字段中，从而在后续分析时可以精确区分"模型生成的错误代码"与"API 调用失败"。

### 3.3.3 对实验可追溯性的意义

上述设计确保了两个关键属性：一是任何实验结果都可以精确追溯到使用的模型名称和调用参数；二是 API 错误不会污染实验数据——分析时可以通过 `api_error` 标记排除受影响的数据点，而非将 API 失败误判为模型能力不足。在本文的全部实验中，正式实验的 API 错误率均为零，验证了该机制的有效性。

## 3.4 Verilog 代码提取、编译与仿真执行

### 3.4.1 代码提取

大语言模型的回复通常包含自然语言解释、Markdown 格式标记和代码块的混合内容。Verilog 提取器的任务是从中可靠地提取出有效的 `module ... endmodule` 代码块。

提取器首先移除 Markdown 代码围栏标记（如 ` ```verilog ` 和 ` ``` `），然后使用正则表达式匹配所有 `module <name> ... endmodule` 结构。当模型回复中包含多个模块定义时，系统取第一个模块作为候选代码。若回复中不包含任何有效模块定义，则该候选将被标记为提取失败（rank = −2.0）。

该设计的一个重要特性是对 CoT（Chain-of-Thought）提示策略的兼容性：当使用 CoT 策略时，模型会在代码之前输出分析文本，提取器能正确跳过这些分析内容，仅提取代码部分。

### 3.4.2 编译执行

编译阶段使用 Icarus Verilog（iverilog）将生成的设计代码与测试平台一起编译。系统在临时目录中创建设计文件，然后调用 `iverilog -g2012` 进行编译，其中 `-g2012` 启用 SystemVerilog 2012 标准支持。编译设置了 30 秒超时保护，防止异常代码导致编译器挂起。

编译结果被封装为 CompileResult 结构，包含编译是否成功、是否有警告、标准输出和标准错误输出。编译成功时，生成的仿真可执行文件路径也被记录，供后续仿真阶段使用。

### 3.4.3 仿真执行与结果解析

仿真阶段使用 VVP 执行编译产生的仿真二进制文件，并设置 60 秒超时保护。仿真完成后，系统需要解析输出以判断测试是否通过。

由于本系统同时支持 VerilogEval 和 RTLLM 两种 benchmark，仿真输出解析器实现了双格式支持：

- **VerilogEval 格式**：匹配 `mismatched samples is N out of M` 模式，其中 N 为不匹配样本数，M 为总样本数。当 N = 0 时判定为通过。
- **RTLLM 格式**：匹配 `Your Design Passed`（直接通过）、`Test completed with N/M failures`（N = 0 时通过）或 `Error`（直接失败）。

仿真结果被封装为 SimResult 结构，包含是否通过、总样本数、不匹配数以及完整的标准输出和标准错误输出。完整的输出信息将用于反馈阶段向模型描述错误细节。

### 3.4.4 评分机制

评分器基于 AutoChip 论文定义的排序规则，将编译与仿真结果映射为一个标量分数：

| 情况 | 分数 | 含义 |
|------|------|------|
| 未提取到有效 Verilog | −2.0 | 代码提取失败 |
| 编译失败 | −1.0 | 存在语法或结构错误 |
| 编译有警告但无仿真结果 | −0.5 | 可能存在问题 |
| 编译成功但无仿真数据 | 0.0 | 无法评估正确性 |
| 仿真部分通过 | 0 ~ 1.0 | (总样本 − 不匹配) / 总样本 |
| 仿真完全通过 | 1.0 | 设计正确 |

当分数达到 1.0 时，反馈循环终止并判定该任务通过。这一连续的评分值也用于在多候选生成时选择最优候选——在每轮迭代的 k 个候选中，系统选择分数最高的候选作为该轮的代表结果。

## 3.5 反馈循环控制机制

反馈循环是本系统的核心创新点，实现了 AutoChip 论文提出的"生成—验证—反馈—修复"迭代闭环。

[此处插入图 3.2：反馈循环流程图（来源：report/figures/fig_feedback_loop_flow.png）]

### 3.5.1 单轮反馈循环

单轮反馈循环（Single-turn Feedback Loop）的执行流程如下：

**第 1 轮**：使用初始提示词生成 k 个候选代码，分别编译、仿真、评分，选出全局最优候选。若最优候选分数为 1.0，则循环终止。

**第 2 轮及后续**：将全局最优候选的代码和对应的编译/仿真错误信息组织为反馈提示词，再次生成 k 个候选。若新候选中有优于当前全局最优的，则更新全局最优。循环直到某候选通过（rank = 1.0）或达到最大迭代次数（默认 5 轮）。

全局最优追踪机制确保了反馈循环具有单调性——每轮的反馈提示词始终基于历史最佳代码及其对应的错误信息构建，即使当前轮生成的代码质量暂时回退，也不会丢失历史最优解。

### 3.5.2 Zero-shot 作为退化形式

零样本（zero-shot）模式可视为反馈循环的退化形式：k = 1、max_iterations = 1，即仅生成一个候选且不进行任何迭代。这一设计使得零样本基线实验和反馈实验共享相同的代码路径，确保了实验条件的公平性。

### 3.5.3 Retry-only 模式

为了区分反馈循环中"多次重试"和"错误反馈信息"各自的贡献，系统实现了 retry-only 模式：在该模式下，所有迭代均使用初始提示词，不包含任何错误反馈信息。与反馈模式使用完全相同的 API 调用预算（k × max_iterations），唯一的区别在于是否包含反馈信息。这一设计使得消融实验中的收益分解成为可能。

### 3.5.4 API 错误处理

当某个候选的 API 调用失败（经过重试后仍然失败）时，该候选被标记为 api_error 但不终止整个迭代。只有当一轮迭代中所有 k 个候选全部因 API 错误而失败时，循环才提前终止并在结果中标记 api_error = True。这种容错设计确保了批量实验的鲁棒性。

## 3.6 反馈策略扩展实现

### 3.6.1 反馈粒度级别

为了研究反馈信息量对修复效果的影响，系统实现了三种反馈粒度级别：

**Level 2 — Compile-only（仅编译反馈）**：仅反馈编译阶段的错误信息。若编译成功但仿真失败，则仅告知"仿真未通过"，不提供任何仿真输出细节。这一级别测试的是：最基础的编译错误信息是否已足够驱动修复。

**Level 3 — Succinct（简洁反馈）**：在编译反馈的基础上，增加仿真失败时的不匹配样本数和仿真输出的前 40 行。这是系统的默认反馈级别，也是所有正式实验和消融实验使用的级别。

**Level 4 — Rich（丰富反馈）**：在简洁反馈基础上进一步扩展：仿真输出增加到前 80 行，额外包含仿真的标准错误输出，并添加显式分析提示（如"检查状态机转移是否正确""验证信号位宽是否匹配"等）。

结合 loop_runner 中 zero-shot（Level 0，无反馈）和 retry-only（Level 1，无反馈但多次重试），系统共支持五个梯度的反馈水平（L0–L4），构成了完整的反馈粒度实验矩阵。

### 3.6.2 多轮对话反馈

除单轮反馈外，系统还实现了多轮对话反馈模式（Multi-turn Feedback）。与单轮模式中每次 API 调用均为独立请求不同，多轮模式维护一个持续增长的对话历史，每次调用将完整的对话上下文发送给模型。

多轮对话的关键设计考虑：（1）初始消息使用简化版提示词，因为后续对话不需要重复任务描述；（2）反馈消息不包含上一轮的代码——因为代码已经存在于对话历史中的 assistant 回复里；（3）反馈信息必须描述模型在对话中最后一轮生成的代码（而非全局最优代码）的错误，确保反馈与上下文一致。

第三点在系统开发过程中曾导致过实验 bug：早期版本错误地使用全局最优结果的编译/仿真信息构建反馈消息，导致当全局最优与最后一轮生成的代码不同时，模型收到的反馈描述的是它未曾见过的代码的错误。该 bug 在项目审计中被发现并修正，修正后的版本使用 `last_iter_comp` 和 `last_iter_sim` 追踪最后一轮的结果，确保反馈一致性。

## 3.7 提示词策略扩展实现

### 3.7.1 四种提示策略

为了研究初始提示词工程对 RTL 生成质量的影响，系统实现了四种可切换的提示策略：

**P0 — Base Prompt（基础提示）**：直接指令式提示，要求模型根据描述和接口生成 Verilog 代码，不输出解释。这是系统的默认策略，也是所有正式实验使用的策略。

**P1 — CoT Prompt（思维链提示）**：要求模型在编码前先分析逻辑类型（组合/时序/FSM）、关键信号关系和时序要求，然后再输出代码。该策略受 Chain-of-Thought 方法启发，测试的是"先推理再编码"是否能提升代码质量。

**P2 — Few-shot Prompt（少样本提示）**：在提示词中提供两个已解决的简单 Verilog 示例（8 位加法器和 4 位计数器），作为上下文学习的参考，然后要求模型生成目标代码。

**P3 — Few-shot + CoT Prompt（少样本 + 思维链）**：结合 P1 和 P2，先提供示例再要求分析后编码。

### 3.7.2 数据泄漏防护

Few-shot 示例的选择需要避免数据泄漏。系统使用的两个示例——8 位加法器（adder_8bit）和 4 位计数器——均不在正式实验使用的 STUDY_12 子集中。加法器来自 RTLLM benchmark 的 CORE_5 子集（打通验证用），计数器为自定义简单任务。两个示例均为基础组合/时序逻辑，不包含目标任务的答案信息。

### 3.7.3 策略调度机制

提示策略通过 PromptStrategy 枚举类型定义，由实验脚本通过 `--prompt-strategy` 命令行参数传入反馈循环控制器。控制器在构建初始提示词时查询策略分发表，选择对应的提示构建函数。反馈阶段的提示词不受策略影响——所有策略在反馈轮次中使用相同的反馈提示模板。策略选择被记录在实验元数据中，确保结果可追溯。

## 3.8 实验产物管理与审计机制

### 3.8.1 实验目录结构

系统采用类型化的实验产物目录结构：`outputs/runs/{category}/{run_id}/`。每个实验运行（run）拥有唯一的 run_id（由前缀和时间戳组成，如 `rtllm_both_20260422_013439`），其目录下包含：

- `summary.json`：汇总结果，包含每个题目的通过/失败状态、rank、API 错误标记等
- `details.json`：完整迭代过程，包含每轮每个候选的提示词、模型回复、提取的代码、编译/仿真输出和评分
- `summary.csv`：表格格式的汇总结果，便于外部工具处理
- `metadata` 信息（嵌入 summary.json 中）：包含 run_id、脚本路径、模型名称、参数配置、Git commit hash、Python 版本和操作系统信息

### 3.8.2 元数据与 Git 集成

每次实验运行的元数据自动采集当前 Git 仓库状态，包括 commit hash、分支名称和工作区是否有未提交的修改。这一设计使得任何实验结果都可以精确关联到产生它的代码版本，在长期项目迭代中避免了"无法复现某次实验结果"的问题。

### 3.8.3 审计脚本

为确保代码正确性和数据一致性，系统配备了两个审计脚本：

**代码审计脚本**（audit_project.py）执行 24 项自动化检查，覆盖评分器逻辑、Verilog 提取器边界情况、仿真输出解析器双格式支持、反馈循环单轮/多轮数据源一致性、提示构建器各粒度级别的信息包含/排除规则以及 retry-only 模式的正确性。

**数据审计脚本**（audit_data.py）遍历所有实验运行目录，从原始 summary.json 中重新计算通过率，与文档记录的结论交叉验证。该脚本覆盖正式实验、消融实验、稳定性实验、粒度实验、多轮对话实验和提示词策略实验共 7 个实验类别。

### 3.8.4 权威实验索引

为防止论文撰写过程中误引用已废弃的实验结果，系统建立了权威实验索引文档。该索引明确标注每个实验阶段的状态（Authoritative / Exploratory / Superseded）、对应的 Git commit、数据目录和分析文档。所有论文和答辩中的数据引用必须经由该索引确认。

## 3.9 本章小结

本章详细介绍了 AutoChip 风格 RTL 自动生成与反馈修复原型系统的设计与实现。系统以统一的 Task 接口屏蔽了不同 benchmark 的格式差异，通过第三方统一 API 网关实现了多模型无缝切换，通过可配置的反馈粒度和提示词策略支持了多维度的实验对比。反馈循环控制器实现了完整的迭代修复闭环，支持单轮反馈、多轮对话、retry-only 等多种实验条件。完善的实验产物管理和审计机制确保了实验结果的可追溯性和可靠性。该系统为第 4 章的实验设计和第 5 章的实验分析提供了坚实的工程基础。
