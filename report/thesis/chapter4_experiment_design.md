# 第 4 章 实验设计

本章定义本文全部实验的目标、评估基准、模型配置、实验条件、评价指标与可靠性控制方法。所有实验设置均来自真实的实验脚本与权威文档，具体实验结果将在第 5 章中呈现和分析。

## 4.1 实验目标与总体设计

### 4.1.1 实验目标

本文的实验旨在系统性地回答以下研究问题：

1. **基础有效性**：AutoChip 风格的 EDA 反馈循环是否能稳定提升大语言模型生成 Verilog 代码的正确性？
2. **反馈价值归因**：反馈循环带来的通过率提升中，有多少来自错误反馈信息本身，有多少仅来自多次重试机会？
3. **结论稳定性**：上述结论在多次独立重复实验下是否稳定，还是受单次采样的随机波动影响？
4. **反馈信息量影响**：不同粒度的反馈信息（从仅编译错误到详细仿真日志）对修复效果有何影响？是否存在"信息过载"效应？
5. **反馈组织方式**：多轮对话式反馈是否优于每轮独立提示的单轮反馈？
6. **提示词策略影响**：初始提示词的工程优化（思维链、少样本示例）对生成质量有多大影响？这些优化与反馈循环是替代关系还是互补关系？
7. **模型依赖性**：上述结论在不同能力水平的模型上是否一致，还是存在显著的模型依赖性？

### 4.1.2 实验总体逻辑

围绕上述研究问题，本文设计了七组递进式实验，实验逻辑如下：

首先，在 VerilogEval-Human 基准上使用 Haiku 模型完成中期基线实验，验证反馈循环的基础有效性（研究问题 1）。

其次，引入更具挑战性的 RTLLM 基准，在 STUDY_12 子集上使用三个不同能力等级的模型进行正式实验矩阵，验证反馈循环在更复杂任务和多模型场景下的表现（研究问题 1、7）。

然后，通过消融实验（研究问题 2）和稳定性重复实验（研究问题 3）回答关于反馈机制的因果归因和统计置信度问题。

最后，通过反馈粒度实验（研究问题 4）、多轮对话实验（研究问题 5）和提示词策略实验（研究问题 6）探索反馈机制的优化方向。

## 4.2 Benchmark 与任务子集

### 4.2.1 VerilogEval-Human 20 题子集

VerilogEval-Human 是 NVlabs 发布的 Verilog 代码生成评估基准的人工编写子集，包含来自 HDLBits 的硬件设计任务。本文在中期阶段选取其中 20 个题目作为基线实验的任务集，覆盖组合逻辑、时序逻辑、有限状态机等多种类型。

选择 VerilogEval-Human 作为中期基线的原因包括：（1）它是学术界广泛使用的 RTL 生成评估基准，便于与已有工作对比；（2）20 题的规模适合快速迭代和验证系统流程；（3）其测试平台采用逐周期比较方式，判定标准明确。

该子集在本文中的角色是中期基础验证实验，用于确认反馈循环在相对简单的任务上的有效性。后续更深入的消融和扩展实验均在 RTLLM 基准上进行。

### 4.2.2 RTLLM 基准与兼容性审计

RTLLM 2.0 是香港科技大学发布的 RTL 设计生成评估基准，包含 50 个涵盖算术运算、控制逻辑、存储器和杂项功能的硬件设计任务。相较于 VerilogEval-Human，RTLLM 的任务普遍更为复杂，涉及 IEEE 754 浮点运算、流水线乘法器、复杂状态机等高难度设计，更能区分模型能力差异和反馈机制的价值。

由于本文使用 Icarus Verilog 作为编译仿真工具，而 RTLLM 中的部分测试平台使用了 iverilog 不完全支持的 SystemVerilog 特性，在接入前进行了全量兼容性审计。审计脚本使用每个题目的参考设计与测试平台一起编译和仿真，结果显示：

| 指标 | 数量 | 比例 |
|------|------|------|
| 总设计数 | 50 | 100% |
| 编译通过 | 46 | 92% |
| 仿真通过 | 41 | 82% |
| 编译失败 | 4 | 8% |
| 编译通过但仿真异常 | 5 | 10% |

编译失败的 4 题中，2 题是测试平台引用了未定义的子模块，2 题是使用了 iverilog 不支持的 SystemVerilog 语法（如 `break` 语句）。仿真异常的 5 题中，3 题是测试平台依赖外部数据文件（`$readmemh`），2 题是参考设计与测试平台预期不一致。

基于该审计结论，本文仅从 41 个完全可用的设计中选取研究子集，确保实验结果不受工具链兼容性问题的干扰。

### 4.2.3 RTLLM_CORE_5 打通验证子集

在进入正式实验前，本文先定义了一个 5 题的最小验证子集 RTLLM_CORE_5，用于端到端验证 RTLLM 加载器、执行器和实验产物体系能否在 RTLLM 上正确工作。CORE_5 的 5 个题目覆盖四大类别（Arithmetic、Control、Memory、Miscellaneous），包含简单到中等难度的任务。

CORE_5 的结果不作为论文正式结论的数据来源，其在权威实验索引中被标记为 Exploratory。

### 4.2.4 RTLLM_STUDY_12 正式研究子集

RTLLM_STUDY_12 是本文后中期阶段所有正式实验、消融实验、稳定性实验、粒度实验、多轮对话实验和提示词策略实验的核心任务集，包含 12 个从 41 个可用设计中精心选取的题目。

## 4.3 模型设置

### 4.3.1 模型接入方式

如第 3 章所述，本系统通过第三方统一 API 网关接入多种大语言模型，保持 API 密钥和服务端点不变，仅通过模型名称参数切换模型。模型名称通过 `LLM_MODEL` 环境变量或实验脚本的 `--model` 命令行参数指定。

### 4.3.2 实验模型选择

本文的实验涉及三个主要模型，代表三个不同的能力等级：

[此处插入表 4.2：实验模型与角色说明]

| 模型名称 | 提供商 | 角色 | 实验覆盖 |
|---------|--------|------|---------|
| claude-haiku-4-5-20251001 | Anthropic | 较弱基线 | VerilogEval 中期基线 + RTLLM 正式实验 |
| claude-sonnet-4-6 | Anthropic | 中强模型 | RTLLM 正式实验 + 消融 + 稳定性 + 粒度 + 多轮 |

选择这三个模型的理由是：它们覆盖了从较弱到强的能力梯度，使得实验结论能够反映反馈机制在不同模型能力水平下的表现差异。Haiku 作为较弱模型验证反馈对弱模型的效果，Sonnet 作为中强模型提供对照，GPT-5.4 作为强模型验证反馈对强模型的效果。

## 4.4 实验条件与对照设计

### 4.4.1 正式实验：Zero-shot 与 Feedback

正式实验矩阵是本文最基础的实验，用于验证反馈循环的有效性。该实验在 RTLLM_STUDY_12 上对三个模型分别运行两种条件：

- **Zero-shot（ZS）**：仅生成一个候选（k = 1），不进行迭代，不提供任何反馈。该条件衡量模型的基础 RTL 生成能力。
- **Feedback（FB）**：生成 k = 3 个候选，最多迭代 5 轮，使用默认的 L3 Succinct 反馈。该条件衡量反馈循环的完整效果。

两种条件的差异包括候选数量、迭代次数和是否有反馈信息。该实验回答研究问题 1（基础有效性）和研究问题 7（模型依赖性）。

### 4.4.2 消融实验：Retry-only 条件

为了区分反馈循环中"多次重试机会"和"错误反馈信息"各自的贡献，设计了三条件消融实验：

| 条件 | 候选数 k | 迭代次数 | 反馈信息 | API 调用预算 |
|------|---------|---------|---------|-------------|
| Zero-shot（ZS） | 1 | 1 | 无 | 1 |
| Retry-only（RO） | 3 | 5 | **无** | 最多 15 |
| Feedback（FB） | 3 | 5 | L3 Succinct | 最多 15 |

Retry-only 与 Feedback 使用完全相同的 API 调用预算，唯一区别在于 Retry-only 的每轮迭代均使用初始提示词，不包含任何编译/仿真错误信息。因此，Feedback 相对于 Retry-only 的增益可以归因于错误反馈信息本身的价值。

该消融实验在 GPT-5.4 和 Sonnet 4.6 上各运行一次。

### 4.4.3 稳定性重复实验

由于大语言模型在 temperature = 0.7 下的输出具有固有随机性，单次实验的结果可能受到偶然因素影响。为验证核心结论的统计稳定性，对消融实验进行了独立重复：

- GPT-5.4：4 轮独立重复（含消融实验的首轮）
- Sonnet 4.6：4 轮独立重复（含消融实验的首轮）

每轮重复是完全独立的 API 调用，不复用任何缓存或历史结果。通过计算 4 轮结果的均值和标准差，可以评估核心结论的稳健性。

### 4.4.4 反馈粒度实验

为探究反馈信息量对修复效果的影响，设计了五级反馈粒度实验：

| 级别 | 名称 | 反馈内容 | 说明 |
|------|------|---------|------|
| L0 | Zero-shot | 无反馈 | 单次生成基线 |
| L1 | Retry-only | 无反馈 | 多次独立重试 |
| L2 | Compile-only | 仅编译错误 | 仿真失败仅告知"未通过" |
| L3 | Succinct | 编译错误 + 仿真摘要 | 默认方法，含不匹配数和输出前 40 行 |
| L4 | Rich | 编译错误 + 详细仿真 + 分析提示 | 仿真输出前 80 行 + stderr + 显式提示 |

L0 和 L1 在 loop_runner 层面控制（不调用反馈提示构建），L2–L4 通过 prompt_builder 的 FeedbackMode 参数控制。五个级别的 API 调用预算相同（L0 除外），从而构成一条从"完全无反馈"到"尽可能丰富反馈"的连续梯度。

该实验在 GPT-5.4 和 Sonnet 4.6 上各运行一次，每个模型运行全部五个级别。

### 4.4.5 多轮对话反馈实验

为比较反馈的组织方式对修复效果的影响，设计了四条件对照实验：

| 条件 | 对话模式 | 反馈级别 | 候选数 k | 设计目的 |
|------|---------|---------|---------|---------|
| A: ST k=3 | Single-turn | L3 Succinct | 3 | 当前默认方法基线 |
| D: ST k=1 | Single-turn | L3 Succinct | 1 | 控制 k 的影响 |
| B: MT k=1 | Multi-turn | L3 Succinct | 1 | 多轮对话模式 |
| C: CO k=3 | Single-turn | L2 Compile-only | 3 | 机制对照 |

其中，条件 D 的引入是为了解耦"对话模式"和"候选数量"两个变量。通过 D 与 B 的对比（均为 k = 1，仅对话模式不同），可以纯粹测量多轮对话的上下文持续效应。通过 A 与 D 的对比，可以测量候选数量（k = 3 vs k = 1）的独立效应。

该实验在 STUDY_12 的一个 6 题代表性子集（Multiturn-6）上运行，以控制实验成本。6 题选择标准为：覆盖不同难度和类别，且包含 feedback 有效的题目和天花板题。

需要说明的是，本实验的初始版本（v1）存在代码 bug——反馈消息使用了全局最优结果而非当前轮结果的编译/仿真信息，导致反馈与对话上下文不匹配。该 bug 在项目审计中被发现并修正。本文报告的所有多轮对话结果均来自修正后的 v2 版本。v1 结果在权威实验索引中被标记为 Superseded，不可作为论文结论引用。

### 4.4.6 提示词策略实验

为研究初始提示词工程与反馈循环的关系，设计了四种提示策略的对比实验：

| 策略 | 名称 | 核心思想 |
|------|------|---------|
| P0 | Base | 直接指令式，要求输出 Verilog 代码 |
| P1 | CoT | 先分析逻辑类型和信号关系，再编码 |
| P2 | Few-shot | 提供两个简单 Verilog 示例作为参考 |
| P3 | Few-shot + CoT | 结合 P1 和 P2 |

每种策略均在 STUDY_12 上运行 Zero-shot 和 Feedback 两种条件，产生 4 × 2 = 8 个实验条件。该实验使用 GPT-5.4 模型。

**数据泄漏防护**：Few-shot 示例选自 RTLLM benchmark 中不在 STUDY_12 子集内的题目（8 位加法器 adder_8bit）和自定义简单任务（4 位计数器），确保示例不包含目标任务的答案信息。

## 4.5 评价指标与统计方法

### 4.5.1 主要评价指标

**通过率（Pass Rate）** 是本文的首要评价指标，定义为在给定条件下通过测试平台全部测试用例的任务数占总任务数的比例。一个任务被判定为通过当且仅当其 rank 分数达到 1.0。

选择通过率作为主指标的原因是：在硬件设计中，部分正确的代码通常不可用——一个有功能错误的 Verilog 模块在实际工程中不能被接受。因此，二值化的通过/失败判定比连续的相似度分数更符合实际需求。

**Rank 分数**作为辅助指标保留，用于在连续尺度上衡量代码质量。其取值范围为 [−2.0, 1.0]，具体定义见第 3 章。Rank 分数主要用于反馈循环内部的候选选择，以及逐题分析中区分"编译失败"与"仿真部分通过"等不同失败模式。

### 4.5.2 派生指标

- **Feedback 改善题数**：在 Feedback 条件下通过但在 Zero-shot 条件下未通过的任务数量，衡量反馈循环的净贡献。
- **API 错误率**：因 API 调用失败（非模型生成失败）而无法获得有效结果的数据点比例。该指标用于评估实验数据的完整性。

### 4.5.3 统计方法

对于稳定性重复实验，本文报告 4 轮重复的均值和标准差。由于 4 次重复的统计效力有限，本文在解读时关注以下两个层面：（1）某一结论是否在所有重复中一致成立（如"FB > RO 是否在 4/4 轮中均成立"）；（2）均值差异是否远大于标准差（信噪比）。

对于单次运行的实验，本文在结论表述中注明其局限性，避免将单次观察过度推广。

## 4.6 实验产物记录与可靠性控制

### 4.6.1 实验产物结构

如第 3 章所述，每个实验运行在 `outputs/runs/{category}/{run_id}/` 目录下生成 summary.json、details.json 和 summary.csv 三类产物文件。summary.json 中嵌入的元数据记录了运行时的模型名称、参数配置、Git commit hash 和运行环境信息，确保每个实验结果可精确追溯到产生它的代码版本。

### 4.6.2 审计与验证机制

为确保实验结果的可靠性，本文建立了两层审计机制：

**代码审计**（audit_project.py）验证系统核心模块的正确性，包含 24 项自动化检查，覆盖评分逻辑、提取器边界情况、仿真解析器、反馈数据源一致性等关键路径。

**数据审计**（audit_data.py）从原始 summary.json 文件中独立重算通过率，与文档中记录的结论交叉验证。该脚本覆盖全部 7 个实验类别的数据。

两项审计均已通过，未发现不一致。

### 4.6.3 权威实验索引

为防止论文撰写中误引用已废弃的历史结果，本文建立了权威实验索引（authoritative_experiment_index.md），对每个实验阶段标注状态（Authoritative / Exploratory / Superseded）、对应的 Git commit、数据目录和分析文档。特别地，多轮对话实验 v1（存在反馈数据源 bug）已被明确标记为 Superseded，其结论不可作为论文结论引用。

本章系统地定义了本文全部实验的设计方案。实验以 VerilogEval-Human 和 RTLLM_STUDY_12 为评估基准，使用三个不同能力等级的大语言模型，通过七组递进式实验回答关于反馈机制有效性、价值归因、信息量影响、组织方式和提示词策略的研究问题。通过稳定性重复实验和完善的审计机制，确保实验结论具备统计稳健性和数据可靠性。这些设计为第 5 章的实验结果分析奠定了方法论基础。
