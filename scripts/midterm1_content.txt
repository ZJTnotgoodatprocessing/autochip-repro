# 本科毕业设计中期报告

**课题名称：** 基于大语言模型的 Verilog 代码自动生成与反馈优化研究

---

## 一、课题总体任务概述

本毕业设计的课题任务是复现并验证 Thakur 等人提出的 AutoChip 框架，该框架利用大语言模型（LLM）结合编译/仿真反馈循环，实现 Verilog 硬件描述语言代码的自动生成与迭代优化。在完成基本复现的基础上，进一步对反馈机制的有效性和局限性进行实验分析，并探索可行的改进方向。

**课题背景。** 随着集成电路设计复杂度的持续增长，手工编写 Verilog 代码面临设计周期长、调试成本高等挑战。Verilog 作为业界最广泛使用的硬件描述语言之一，其编写需要设计者同时具备数字电路理论知识和丰富的工程经验。近年来，以 GPT-4、Claude 等为代表的大语言模型在代码生成领域取得了显著进展，研究者开始探索将其应用于硬件设计领域，尝试让模型根据自然语言的功能描述自动生成 Verilog 代码。然而，与软件代码不同，Verilog 代码的正确性要求更为严格：不仅需要通过编译，还需要在功能仿真中与预期行为完全一致（即所有测试向量均通过）。仅靠 LLM 一次性生成往往难以保证正确性，这就需要引入额外的验证和修复机制。

**AutoChip 的核心思路。** AutoChip 框架的核心创新在于引入了一个闭环反馈机制：LLM 首先根据自然语言描述和模块接口定义生成 Verilog 代码，随后系统自动调用编译器（iverilog）和仿真器（vvp）对代码进行验证。如果验证未通过，系统将编译错误信息或仿真失败的不匹配统计反馈给 LLM，引导其进行针对性修复。这一过程可以反复迭代，直至代码通过所有测试或达到最大迭代次数。简单来说，这模拟了工程师"写代码、编译、看报错、改代码"的日常工作流程，只不过"写代码"和"改代码"的环节由 LLM 完成。该框架还采用了多候选生成策略（每轮生成 k 个候选，选最优）和精简反馈策略（只提取关键错误信息，控制 Token 消耗），以提高迭代效率。原论文的实验表明，这种反馈机制能够将 LLM 生成正确 Verilog 代码的成功率提升数十个百分点。

**评测基准。** 本研究采用 NVlabs 发布的 VerilogEval-Human 基准测试集作为评测标准。该测试集源自 HDLBits 在线练习平台，包含 156 道由人类专家编写的 Verilog 设计题目，涵盖组合逻辑、时序逻辑、有限状态机等多种电路类型。每道题目配有完整的 SystemVerilog 测试平台和金标准参考实现，能够自动统计设计模块与参考模型之间的不匹配样本数，提供客观、可量化的评测结果。

**本课题的具体任务包括：** （1）独立实现 AutoChip 的完整流水线，包括任务加载、LLM 调用、Verilog 提取、编译、仿真、评分和反馈循环；（2）接入 VerilogEval-Human 标准基准测试集，在规范化的题目上进行实验；（3）设计对照实验，比较零样本生成与反馈循环的效果差异；（4）分析反馈机制的适用范围和局限性；（5）在复现基础上探索可行的改进方向。

## 二、目前已完成的工作与阶段性成果

截至中期检查，本课题已从系统搭建阶段进入实验分析阶段。以下按时间顺序梳理已完成的各项工作。

### 2.1 实验环境搭建与流水线验证（已完成）

项目基于 Windows 11 操作系统、Python 3.12 开发，使用 Icarus Verilog（iverilog）作为 Verilog 编译器、vvp 作为仿真执行器。LLM 后端采用 Anthropic 的 Claude 系列模型，通过第三方 API 中继服务进行调用。项目使用 Git 进行版本控制，所有实验结果以 JSON 和 CSV 格式保存，确保可追溯性。

环境搭建完成后，首先进行了端到端的流水线验证。使用一个最简单的任务（wire 直连：将输入信号直接连接到输出）测试了完整的流程：任务加载、LLM 调用、Verilog 代码提取、iverilog 编译、vvp 仿真、结果评分。验证结果表明流水线各环节工作正常，LLM 能够正确理解任务描述并生成可编译、可仿真的 Verilog 代码。

### 2.2 自定义任务集构建与初步验证（已完成）

为了在接入标准基准测试之前充分验证系统功能，手工构建了 9 个自定义 Verilog 设计任务，涵盖不同的电路类型和难度级别。其中 5 个为基础任务（wire 直连、AND 门、2:1 多路选择器、8 位加法器、population count），4 个为进阶任务（上升沿检测器、8 位优先编码器、带溢出检测的有符号加法器、5 位 LFSR）。每个任务包含自然语言功能描述、模块接口定义和测试平台三个文件，测试平台采用与 VerilogEval 兼容的输出格式。

使用 Claude Opus 模型对 9 个自定义任务进行了零样本批量测试，结果为 9/9 全部通过（通过率 100%）。这一结果验证了系统流水线的正确性，但也表明自定义任务对于高能力模型而言过于简单，无法体现反馈循环的价值。这一发现促使本研究转向更具挑战性的标准基准测试集。

### 2.3 反馈循环核心机制实现（已完成）

反馈循环是 AutoChip 框架的核心，也是本复现工作的重点。实现过程中新增了以下关键模块：

**提示词构建器**（prompt_builder.py）：实现了零样本提示词和反馈提示词的构建。零样本提示词将任务描述和模块接口组装为 LLM 输入；反馈提示词使用预定义模板，将原始任务信息、上一轮最佳代码和精简的错误反馈信息填充其中。反馈信息的精简逻辑会根据错误类型提取关键信息：编译失败时截取前 30 行编译错误，仿真失败时截取不匹配统计和前 40 行仿真输出。这种精简策略参考了原论文的设计，在控制 Token 消耗的同时保留了足够的错误定位信息。

**反馈循环控制器**（loop_runner.py）：实现了完整的迭代控制逻辑。每轮迭代中，控制器调用 LLM 生成 k 个候选代码，对每个候选依次执行提取、编译、仿真、评分流程，选出本轮最佳候选，并与全局最优进行比较更新。系统采用连续的 0 到 1.0 评分标度（基于仿真匹配率），而非简单的通过/失败判断，使得即使代码尚未完全通过，也能追踪质量的渐进改善。控制器还实现了 API 错误处理机制，单个候选的 API 调用失败不会中断整个循环。

**LLM 客户端增强**（client.py）：针对第三方 API 中继服务的不稳定性，实现了指数退避重试机制。系统自动识别可重试的瞬态错误（HTTP 429/500/502/503/504/529 状态码、连接超时等），在 2 秒、5 秒、10 秒的递增间隔后重试，最多重试 3 次。同时实现了错误分类功能，便于实验结果的统计分析。

### 2.4 VerilogEval-Human 基准测试接入（已完成）

接入 VerilogEval-Human 基准测试集的过程中，解决了以下技术适配问题：（1）VerilogEval 的测试平台使用了 SystemVerilog 特性，需要在 iverilog 编译命令中添加 -g2012 标志以启用 SystemVerilog 2012 支持；（2）VerilogEval 的测试平台同时引用被测设计模块（TopModule）和金标准参考模块（RefModule），后者存储在独立文件中，本研究采用文件合并策略将二者拼接后统一编译；（3）VerilogEval 使用 TopModule 作为模块名，而自定义任务使用 top_module，由于提示词构建器直接使用任务的模块接口声明，LLM 会自然生成与接口一致的模块名，无需额外适配；（4）新增了 verilogeval_loader.py 适配器模块，负责从 VerilogEval 仓库读取题目信息并转换为系统内部的数据结构。

从 156 题中选取了 20 道作为实验子集，覆盖 Easy（8 题）、Medium（7 题）、Medium-Hard（1 题）、Hard（3 题）和 Very Hard（1 题）五个难度级别，涵盖组合逻辑、时序逻辑和有限状态机三种主要电路类型。

### 2.5 主实验执行（已完成）

使用 Claude Haiku（claude-haiku-4-5-20251001）模型在 20 题子集上完成了零样本与反馈循环的对照实验。选择 Haiku 而非更强大的 Opus 模型的原因是：预实验中 Opus 在 20 题上零样本通过率即达到 100%（20/20），存在天花板效应，无法体现反馈循环的改善效果。Haiku 作为较小的模型，零样本能力存在明显的提升空间，更适合用于验证反馈机制的有效性。实验参数设置为：温度 0.7，零样本条件 k=1、不迭代，反馈循环条件 k=3、最多迭代 5 轮。实验过程中未出现 API 错误，20 道题目均获得了有效结果。

**阶段性成果小结：** 目前已完成从环境搭建、核心算法实现、基准测试接入到主实验执行的全部工作，系统能够稳定运行并产出可分析的实验数据。项目已进入实验结果分析和后续实验设计阶段。

## 三、当前实验结果与初步分析

### 3.1 总体结果

使用 Claude Haiku（claude-haiku-4-5-20251001）模型在 20 题子集上完成了零样本与反馈循环的对照实验。选择 Haiku 而非更强大的 Opus 模型，是因为预实验中 Opus 零样本通过率即达到 100%，存在天花板效应，无法体现反馈循环的改善空间。实验参数设置为：温度 0.7，零样本条件 k=1、不迭代，反馈循环条件 k=3、最多迭代 5 轮。

汇总结果如下：

| 方法 | 通过数 | 总数 | 通过率 |
|------|--------|------|--------|

### 3.3 四道关键题目的案例分析

以下对 4 道零样本未通过的题目进行逐一分析，以揭示反馈循环的适用场景和局限性。

**Prob109_fsm1（简单 FSM，反馈修复成功）。** 零样本生成的代码 rank=0.0%，仿真完全不匹配。反馈循环在第 2 轮通过编译/仿真错误信息定位了状态转移条件的问题，成功修正并通过全部测试。这体现了反馈循环对局部逻辑错误的修复能力。

**Prob127_lemmings1（Lemmings 游戏 FSM，反馈修复成功）。** 零样本 rank=63.8%，约 36% 的测试样本不匹配。该状态机需要处理行走、坠落、方向切换等多个状态，零样本代码在部分边界状态转移上有误。反馈循环利用仿真的不匹配信息，在第 2 轮即修复了剩余错误。这是反馈循环最典型的成功场景：代码已"接近正确"，只需少量针对性修正。

**Prob140_fsm_hdlc（HDLC 协议 FSM，反馈未能修复）。** 零样本 rank=92.5%，反馈循环经过 5 轮迭代仅提升至 97.1%。HDLC 帧检测需要精确识别 flag 序列、bit stuffing 和 abort 条件，剩余约 3% 的错误可能涉及极端边界条件，这些场景在仿真错误信息中难以直接定位。这说明反馈循环对需要全局协议理解的深层逻辑错误修复能力有限。

**Prob082_lfsr32（32 位 LFSR，反馈未能修复）。** 零样本 rank=0.02%，反馈循环 5 轮迭代后仍为 0.02%，无任何改善。32 位线性反馈移位寄存器需要精确的抽头多项式，如果多项式错误则输出序列完全不同。反馈循环无法从"几乎全部不匹配"的仿真结果中推断出正确的抽头位置。这揭示了反馈循环的根本局限：当错误源于缺失的领域知识（特征多项式）而非代码逻辑时，迭代修复无效。

### 3.4 初步结论

基于上述实验结果，可以得出以下初步结论：（1）反馈循环能够有效提升 LLM 生成 Verilog 代码的通过率，本实验中从 80% 提升至 90%；（2）反馈循环对"接近正确"的代码最为有效，尤其是存在局部状态转移错误的 FSM 题目；（3）反馈循环对需要精确领域知识的问题（如 LFSR 抽头多项式）和需要全局协议理解的复杂状态机（如 HDLC）效果有限；（4）Easy/Medium 级别题目 Haiku 模型零样本即可解决，反馈循环的价值主要体现在 Hard 及以上难度。

需要指出的是，以上结论基于 20 题子集和单一模型的实验，结论的普适性有待后续在更大规模实验中验证。

## 四、当前存在的问题与不足

在研究推进过程中，目前存在以下问题和不足：

**1. 实验规模有限。** 当前仅在 VerilogEval-Human 的 20 题子集上进行了实验，占完整测试集（156 题）的约 13%。子集的选取虽然覆盖了 Easy 到 Very Hard 五个难度级别和组合逻辑、时序逻辑、有限状态机三种电路类型，但样本量较小，个别题目的结果可能受随机因素影响，统计结论的可靠性和普适性受限。

**2. 模型覆盖不足。** 目前仅使用了 Claude Haiku 一个模型进行主实验。原论文测试了 GPT-4、GPT-3.5、CodeLlama 等多个模型，不同模型的零样本能力和对反馈的响应特性可能存在显著差异。缺乏多模型对比，难以评估反馈机制的通用性。

**3. 反馈信息利用方式单一。** 当前的反馈策略仅将编译错误或仿真不匹配统计以文本形式反馈给 LLM，属于单轮反馈。未探索更精细的反馈形式，如逐信号波形对比、错误时间点定位等，可能限制了反馈的信息量。

**4. API 调用依赖第三方中继。** 由于网络环境限制，LLM 调用通过第三方 API 中继服务进行，存在延迟波动和偶发不可用的问题。虽然已实现了指数退避重试机制，但在大规模实验中仍可能影响实验效率。

**5. 缺乏与原论文的严格对比条件。** 原论文使用 GPT-4 等模型在完整 156 题上实验，本研究使用不同模型和题目子集，因此无法进行严格的数值对比，只能在方法论层面进行定性比较。

## 五、下一阶段研究计划

后续工作将围绕以下几个方面展开：

**1. 扩大实验规模。** 将实验范围从 20 题子集扩展至 VerilogEval-Human 完整的 156 题，以获得更具统计意义的结果。这需要解决大规模实验的 API 调用成本和时间管理问题。

**2. 增加模型对比。** 在条件允许的情况下，增加至少一个不同能力级别的模型（如 Claude Sonnet）进行对照实验，观察反馈循环在不同模型上的效果差异。

**3. 完善实验分析。** 对实验结果进行更细致的分类统计，按难度级别、电路类型分别计算反馈循环的提升效果，分析反馈循环在哪些类别的题目上最为有效。

**4. 优化反馈策略。** 尝试改进反馈信息的构建方式，例如提取更精确的错误定位信息、调整反馈模板的措辞，观察是否能进一步提升反馈循环的修复成功率。

**5. 撰写毕业论文。** 在实验完成后，整理所有实验数据和分析结论，按照毕业论文规范撰写完整的毕业设计论文。

## 六、改进方向与研究思路

在完成基本复现和实验分析的基础上，本研究拟探索以下改进方向：

**方向一：反馈信息精细化。** 当前反馈仅包含编译错误文本或仿真不匹配统计的摘要，信息粒度较粗。可以尝试提取更结构化的反馈信息，例如将仿真波形中首次出现不匹配的时间点和相关信号值提供给 LLM，帮助其更精确地定位错误根源。此外，还可以尝试区分不同类型的错误（如组合逻辑错误与时序逻辑错误），为 LLM 提供更有针对性的修复建议。这一方向的技术可行性较高，可在现有框架上直接扩展。

**方向二：提示词工程优化。** 探索不同的提示词构建策略对生成质量的影响。例如，在提示词中加入相关的 Verilog 设计模式示例（few-shot），或者针对不同电路类型使用不同的提示词模板。这一方向实现成本较低，但效果需要通过实验验证。

**可选拓展方向：多轮对话式反馈。** 当前框架中每轮反馈是独立的（LLM 不保留之前的对话历史）。如果改为多轮对话模式，LLM 可以积累之前的修复尝试信息，避免重复犯错。但这一方向会显著增加 Token 消耗，需要权衡成本与收益，视时间和资源情况决定是否实施。

## 七、总结

本毕业设计截至中期检查，已完成 AutoChip 框架的独立复现，包括完整的 LLM 调用、Verilog 编译仿真、评分和反馈循环流水线。系统已成功接入 VerilogEval-Human 标准基准测试集，并在 20 题子集上完成了零样本与反馈循环的对照实验。实验结果表明，反馈循环将 Haiku 模型的通过率从 80% 提升至 90%，验证了反馈机制的有效性，同时也通过案例分析揭示了其在领域知识缺失和复杂协议逻辑方面的局限性。总体而言，项目进展符合预期，核心的复现和验证工作已基本完成。下一阶段将重点扩大实验规模、增加模型对比，并探索反馈策略的优化方向，最终完成毕业论文的撰写。
